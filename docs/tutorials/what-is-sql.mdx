If you're a Flutter developer building a backend with Globe DB, you're using Drift to define tables and run CRUD operations. Under the hood, that generates **SQL** (Structured Query Language). Understanding SQL helps you design better schemas and debug what your code is doing. This tutorial explains what SQL is, how a database relates to tables and rows, and the core ideas you'll see in Globe DB and Drift.

This tutorial will walk you through what SQL is, the relationship between a database and its tables (one database, many tables), how tables relate to each other, data modeling and normalization, primary and foreign keys, basic queries (SELECT, INSERT, UPDATE, DELETE), and JOINs.

## Exploring SQL

### 1. What is SQL?

**SQL** (Structured Query Language) is how you talk to a **relational database**. You use it to create tables, insert and update data, ask questions (queries), and define relationships between tables. When you use Drift in Dart, you define your schema in code; Drift turns that into SQL and runs it against your database.

Globe DB is built on **SQLite**, a lightweight relational database. So when you create a Globe DB and use Drift, you're using SQL under the hood. A bit of SQL literacy makes it easier to reason about your schema and to read logs or errors when something goes wrong.

### 2. Database, Tables, Columns, and Rows

A common confusion is thinking you have one database per table. In fact, **one database holds many tables**.

- **Database**: The container. When you create a Globe DB in the dashboard, you get **one** database (e.g. `my-app-db`). That single database is where all your app's persistent data lives.

- **Tables**: Inside that database you create **tables**. Each table has a name (e.g. `users`, `notes`) and represents one kind of thing. You can have many tables in the same database: for example, one database might contain both a `users` table and a `notes` table.

- **Columns**: Each table has a fixed set of **columns** (the field names and types). For a `notes` table you might have columns like `id`, `title`, `content`, `created_at`. Columns define the *structure* of the table.

- **Rows**: **Rows** are the actual records. Each row has one value per column. So one row in `notes` might be `id: 1, title: "Hello", content: "World", created_at: 2025-01-15`. Add more rows and the table grows; the columns stay the same.

So the hierarchy is: **one database → many tables → each table has columns (structure) and rows (data)**. In Globe DB, you create one database per app (or per environment); inside it you define as many tables as your app needs.

### 3. Tables and Relationships

Tables often need to reference each other. For example, a note might belong to a user. You don't copy the user's name into every note row; instead, you store a **reference** to the user (e.g. a `user_id` column in the `notes` table that points to a row in the `users` table). That's a **relationship**: one user has many notes (one-to-many).

Relationships are enforced with **foreign keys**: a column in one table that holds the primary key of another table. So `notes.user_id` might reference `users.id`. That way, you keep one source of truth for the user and link notes to them. We'll cover primary and foreign keys in a moment.

### 4. Data Modeling

**Data modeling** is deciding what tables you need and what columns each has, based on your app's needs. You think in terms of **entities** (users, notes, products) and **relationships** (user has many notes, product belongs to a category). Then you create one table per entity and add columns for their attributes and for any foreign keys that express the relationships. Getting the model right up front makes queries and updates much simpler later.

### 5. Normalization (1NF, 2NF, 3NF)

**Normalization** is a set of rules for structuring data so you avoid redundancy and inconsistency. In short:

- **1NF (First Normal Form)**: Each column holds a single value; no repeating groups. Rows are unique (e.g. each row has a unique primary key).

- **2NF (Second Normal Form)**: Builds on 1NF. Every non-key column depends on the *whole* primary key, not just part of it. So you don't store the same fact in multiple places.

- **3NF (Third Normal Form)**: Builds on 2NF. Non-key columns don't depend on other non-key columns. So you don't duplicate derived or redundant data.

You don't need to memorize the formal definitions. The idea is: **split data into tables so each fact lives in one place**. When you need to change something (e.g. a user's name), you change it once; relationships (foreign keys) keep everything consistent. That's why we have separate `users` and `notes` tables instead of repeating the user's name in every note row.

### 6. Primary and Foreign Keys

- **Primary key**: A column (or set of columns) that uniquely identifies each row in a table. Usually you use an `id` column (integer, often auto-increment). Every table should have a primary key so you can refer to a specific row.

- **Foreign key**: A column in one table that stores the primary key value of a row in *another* table. For example, `notes.user_id` might be a foreign key to `users.id`. That enforces the relationship: a note "belongs to" a user, and the database can ensure you don't reference a user that doesn't exist.

Together, primary and foreign keys keep your data consistent and let you express relationships (one-to-many, many-to-one) between tables.

### 7. Basic Queries

SQL uses a small set of verbs for the main operations. You'll see these when you use Drift or read SQL logs:

- **SELECT**: Read data. "Select these columns from this table, optionally where some condition holds."

  ```sql
  SELECT id, title, content FROM notes WHERE id = 1;
  ```

- **INSERT**: Create a new row.

  ```sql
  INSERT INTO notes (title, content) VALUES ('Hello', 'World');
  ```

- **UPDATE**: Change existing row(s).

  ```sql
  UPDATE notes SET title = 'Hi' WHERE id = 1;
  ```

- **DELETE**: Remove row(s).

  ```sql
  DELETE FROM notes WHERE id = 1;
  ```

In Drift, you write Dart code that generates SQL like this; understanding SELECT, INSERT, UPDATE, and DELETE helps you map between your Dart API and what the database is doing.

### 8. JOINs

Data you need often lives in **multiple tables**. For example, you want to show a list of notes with each note's author name. The author name is in `users`; the note is in `notes`. A **JOIN** combines rows from two (or more) tables by matching on a key.

For example, to get each note with its author's name, you join `notes` to `users` on `notes.user_id = users.id`:

```sql
SELECT notes.title, notes.content, users.name AS author_name
FROM notes
JOIN users ON notes.user_id = users.id;
```

That returns one row per note, with the corresponding user's name attached. JOINs are how you follow relationships in a single query. Drift supports joins in Dart, so you can fetch related data without writing raw SQL, but the idea is the same: match rows across tables by their keys.
