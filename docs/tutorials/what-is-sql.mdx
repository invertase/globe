If you're building a backend, you'll work with a database. That usually means **SQL** (Structured Query Language). This guide explains the core concepts: how data is stored, how tables link, and how you read and write it.

## Exploring SQL

### 1. Database, Tables, Columns, and Rows

Relational data has a simple hierarchy: **one database** holds **many tables**; each table has **columns** (structure) and **rows** (data). You don't get one database per table.

- **Database**: The container for your stored data. One database holds all your tables.

- **Table**: A named set of records of one kind (e.g. `users`, `notes`). Each table has a fixed set of columns.

- **Columns**: The field names and types (e.g. `id`, `name`, `email`). Columns define the structure of the table.

- **Rows**: The records. Each row has one value per column. Add more rows and the table grows; the columns stay the same. For example, one row in `users` might be: `id` 1, `name` 'Alice', `email` 'alice@example.com'.

#### Primary Key

Every row needs a unique identity so you can find, update, or delete it. The **primary key** is a column (or set of columns) that uniquely identifies each row. Usually an integer `id` that auto-increments. In practice, every table should have one.

### 2. Relationships

In SQL, related data lives in separate tables and is linked by keys; so you keep one source of truth and avoid duplicate or inconsistent data. For example, users and their notes: each note belongs to one user, and we link them so we can query "notes for this user" or "who wrote this note."

#### Why we split data

If you stored the user's name in every note row, and the user changed their name, you'd have to update every note. Instead, we split data into separate tables and link them.

#### Foreign key (the how)

A **foreign key** is a column in one table that stores the primary key of a row in another table. It creates the link.

Example: a `notes` table has `user_id`. That column holds the `id` of a row in the `users` table. So each note points to one user. The database can enforce that the referenced row exists.

### 3. Reading and Writing Data (CRUD)

You talk to the database with **SQL statements**.

**INSERT**: add a new row:

```sql
INSERT INTO notes (title, content, user_id) VALUES ('Buy milk', 'Milk, eggs', 1);
```

The `id` is usually auto-generated by the database, so we don't include it.

**SELECT**: read data. This returns all columns from `notes` where `user_id` is 1:

```sql
SELECT * FROM notes WHERE user_id = 1;
```

**UPDATE**: change existing rows:

```sql
UPDATE notes SET title = 'Buy almond milk' WHERE id = 101;
```

**DELETE**: remove rows:

```sql
DELETE FROM notes WHERE id = 101;
```

### 4. Combining Data (JOINs)

Data you need often sits in more than one table. To get a note with its author's name, you **JOIN** the tables: match rows in `notes` with rows in `users` where the foreign key equals the primary key.

```sql
SELECT notes.title, users.name
FROM notes
JOIN users ON notes.user_id = users.id;
```

You get one result row per note, with the matching user's name attached. This is an **INNER JOIN**: only notes that have a matching user appear.

## Summary

- **Database**: The container; one database holds all your tables.
- **Table**: A set of records with the same columns.
- **Row**: One record (one value per column).
- **Primary key**: Uniquely identifies each row in a table.
- **Foreign key**: A column that references a primary key in another table (the link).
- **JOIN**: Combines rows from two or more tables by matching keys.
